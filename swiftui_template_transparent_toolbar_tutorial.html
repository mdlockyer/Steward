1. Target result

This template is a small macOS SwiftUI app shell for starting new projects without rebuilding the same navigation, toolbar, appearance, settings, project generation, and packaging scaffolding every time. The important visual behavior is the top edge: content scrolls underneath a native transparent toolbar region, and the system supplies the soft scroll-edge transition that keeps toolbar text and controls legible without adding a hard separator.

The app intentionally does not implement the transcript-style bottom gradient/mask from the Turnstile document. The bottom edge here is only a reserved layout zone: a clear safeAreaInset gives scroll content room to clear future bottom chrome, but it does not darken, fade, blur, or tint the content.

2. Architecture: the ownership split

The stable pattern is to let each level own exactly one job. The app entry point owns process and window policy. The split root owns selection and the toolbar host. Each destination screen owns its own scroll view and scroll-edge behavior. The screen registry owns the mapping from sidebar case to destination view.

App shellSwiftUITemplateApp creates the scene, applies appearance, disables unwanted window affordances, and sizes the main window.
Split rootContentView owns sidebar selection, renders NavigationSplitView, and hosts the toolbar picker.
DestinationLoremIpsumScreen and DolorSitScreen own the actual ScrollView and its top scroll-edge effect.
RegistryScreen declares what appears in the sidebar and what view each row opens.

This split is what keeps the top toolbar reliable. A parent wrapper can host the toolbar, but the scroll-edge effect must remain attached to the actual scroll view whose content is moving under the toolbar. Conversely, the toolbar background should not be hidden or replaced by an ad hoc overlay; it should remain system-managed.

3. App shell: launch, scene, appearance, and window policy

The app entry point is deliberately compact, but it does more than create a window. It disables automatic window tabbing, ensures the app activates as a regular macOS app, stores the selected appearance in AppStorage, and uses an NSApplicationDelegate for menu and window cleanup that SwiftUI does not expose cleanly at the scene level.

Sources/App/SwiftUITemplateApp.swiftcore scene and appearance loop
import AppKit
import SwiftUI

@main
struct SwiftUITemplateApp: App {
    @NSApplicationDelegateAdaptor(AppDelegate.self) private var appDelegate
    @AppStorage(SettingsStorageKey.colorSchemeMode.rawValue) private var colorSchemeMode: ColorSchemeMode = .system

    init() {
        NSWindow.allowsAutomaticWindowTabbing = false
        NSApplication.shared.setActivationPolicy(.regular)

        DispatchQueue.main.async {
            NSApp.activate(ignoringOtherApps: true)
        }
    }

    var body: some Scene {
        Window("SwiftUITemplate", id: "main") {
            ContentView(colorSchemeMode: $colorSchemeMode)
                .onAppear { applyColorScheme() }
                .onChange(of: colorSchemeMode) { applyColorScheme() }
        }
        .commands {
            CommandGroup(replacing: .windowArrangement) { }
        }

        Settings {
            SettingsView()
        }
        .defaultSize(width: 560, height: 340)
    }

    private func applyColorScheme() {
        NSApp.appearance = colorSchemeMode.nsAppearance
    }
}

The appearance setting is intentionally applied through NSApp.appearance, not by sprinkling .preferredColorScheme through the view tree. That gives the template a native macOS-wide appearance switch: toolbar material, sidebar chrome, settings window controls, and normal SwiftUI content all resolve against the same AppKit appearance.

Window setup

configureMainWindow() enforces a practical minimum size and centers the first launch window if it is smaller than the intended template baseline.

Menu setup

The delegate removes unsupported menu items so the generated starter app feels intentional rather than like a raw SwiftUI sample.

4. Split root: sidebar selection and detail toolbar host

ContentView is the architectural center of the visual pattern. It owns the selected Screen, builds the sidebar List, renders the selected destination in the detail column, and attaches the toolbar to the detail column rather than to the individual screens.

Sources/Navigation/ContentView.swiftNavigationSplitView + toolbar owner
struct ContentView: View {
    @State private var selection: Screen?
    @Binding var colorSchemeMode: ColorSchemeMode

    init(colorSchemeMode: Binding<ColorSchemeMode>, initialSelection: Screen? = .loremIpsum) {
        self._colorSchemeMode = colorSchemeMode
        self._selection = State(initialValue: initialSelection)
    }

    var body: some View {
        NavigationSplitView {
            List(selection: $selection) {
                Section("Lorem") {
                    ForEach(Screen.allCases) { screen in
                        Label(screen.title, systemImage: screen.systemImage)
                            .tag(screen)
                    }
                }
            }
            .listStyle(.sidebar)
        } detail: {
            detailView
                .toolbar {
                    ToolbarItem(placement: .primaryAction) {
                        Picker("Appearance", selection: $colorSchemeMode) {
                            Image(systemName: "sun.max").tag(ColorSchemeMode.light)
                            Image(systemName: "moon.fill").tag(ColorSchemeMode.dark)
                            Image(systemName: "circle.lefthalf.filled").tag(ColorSchemeMode.system)
                        }
                        .pickerStyle(.segmented)
                        .glassEffect()
                        .labelsHidden()
                        .help("Choose the app appearance")
                    }
                }
                .toolbarBackgroundVisibility(.automatic, for: .windowToolbar)
        }
    }

    @ViewBuilder
    var detailView: some View {
        (selection ?? .loremIpsum).destination
    }
}

The toolbar is intentionally attached after detailView. That gives every destination the same right-side appearance control without making every screen repeat toolbar code. The destinations still own their own .navigationTitle(...), so the title shown in the toolbar remains screen-specific.

5. The top toolbar: why it stays transparent

The transparent toolbar is the result of three pieces cooperating, not one magic modifier. The toolbar must have a system-managed background available, the scroll view underneath it must be visually transparent, and the scroll view must request a top scroll-edge effect.

ResponsibilityOwnerRequired modifierWhy it matters
Toolbar materialContentView detail host.toolbarBackgroundVisibility(.automatic, for: .windowToolbar)Keeps the native window toolbar background available for the system scroll-edge treatment.
Scrollable contentEach destination screenScrollView EllipsisThe system effect must attach to the scroll view that actually moves.
Transparent viewportEach destination screen.scrollContentBackground(.hidden) + .background(.clear)Prevents the scroll view from painting an opaque backing that blocks the toolbar material.
Top soft edgeEach destination screen.scrollEdgeEffectStyle(.soft, for: .top)Requests the native fade/softening where content enters the top safe-area/toolbar region.

The visual model is simple: the toolbar remains a native toolbar; the scroll view remains a native scroll view; the two meet at the top safe-area boundary. The app does not draw a fake toolbar background, does not add a hard separator, and does not flatten the scroll view into a parent mask.

6. Destination screens: the reusable scroll contract

The template ships with two destination shapes: a card-oriented screen and a list-row screen. Both use the same outer scroll recipe. This is the most important convention for future screens in the template: the content can vary, but the scroll chrome contract should stay consistent.

Sources/Screens/LoremIpsumScreen.swiftcard screen recipe
struct LoremIpsumScreen: View {
    var body: some View {
        ScrollView {
            VStack(alignment: .leading, spacing: 14) {
                Text("Lorem Ipsum")
                    .font(.largeTitle)
                    .padding(.bottom, 4)

                ForEach(cards) { card in
                    VStack(alignment: .leading, spacing: 8) {
                        Text(card.title).font(.headline)
                        Text(card.body)
                            .font(.body)
                            .foregroundStyle(.secondary)
                    }
                    .padding(16)
                    .frame(maxWidth: .infinity, alignment: .leading)
                    .background(.background, in: RoundedRectangle(cornerRadius: 10))
                    .overlay(RoundedRectangle(cornerRadius: 10).stroke(.separator, lineWidth: 0.5))
                }
            }
            .padding(24)
        }
        .scrollContentBackground(.hidden)
        .background(.clear)
        .scrollEdgeEffectStyle(.soft, for: .top)
        .safeAreaInset(edge: .bottom, spacing: 0) { Color.clear.frame(height: 56) }
        .navigationTitle("Lorem Ipsum")
    }
}

The practical rule is: when a screen scrolls, copy the scroll recipe before designing the content. That keeps toolbar behavior stable across cards, rows, forms, inspectors, settings-like panes, and future project-specific destinations.

7. Bottom edge: clear inset, not a fade

This template deliberately leaves the bottom edge visually neutral. There is no bottom gradient, no multiply blend, no content mask, and no Liquid Glass composer. The bottom modifier exists for one reason: layout breathing room.

Important distinction: safeAreaInset(edge: .bottom) changes layout. It does not apply a visual treatment. The clear view reserves space so the last row or card can scroll above future bottom chrome instead of being clipped against the window edge.
Bottom inset contractlayout-only spacer
.safeAreaInset(edge: .bottom, spacing: 0) {
    Color.clear.frame(height: 56)
}

If a future app built from this template adds a floating composer, media transport, inspector rail, or bottom command bar, that feature should be implemented as a separate layer with its own ownership. Do not preemptively add a bottom edge effect to the base template; the base template should stay visually quiet and predictable.

8. Screen registry: adding a new destination

The sidebar is driven by Screen.allCases. Adding a new screen should be a registry edit, not a rewrite of the navigation root. This keeps the template fast to adapt: add a case, give it a title and SF Symbol, then return the destination view.

Sources/Navigation/Screen.swiftsidebar registry
enum Screen: String, CaseIterable, Hashable, Identifiable {
    case loremIpsum
    case dolorSit

    var id: Self { self }

    var title: String {
        switch self {
        case .loremIpsum: "Lorem Ipsum"
        case .dolorSit: "Dolor Sit"
        }
    }

    var systemImage: String {
        switch self {
        case .loremIpsum: "snowflake"
        case .dolorSit: "wind.snow"
        }
    }

    @ViewBuilder
    var destination: some View {
        switch self {
        case .loremIpsum: LoremIpsumScreen()
        case .dolorSit: DolorSitScreen()
        }
    }
}
Adding a screenminimal extension
// 1. Add the case.

// 2. Add the title.

// 3. Add the sidebar symbol.
case .dashboard: "rectangle.3.group"

A new destination should usually start from the destination contract in section 6. The navigation root should not need to know whether the screen is card-based, row-based, form-based, or chart-based.

9. Appearance control: toolbar picker, settings pane, and AppStorage

The appearance picker in the toolbar and the appearance picker in Settings share the same AppStorage key. That is the correct template pattern: two controls, one persisted setting, one application-level effect.

Sources/Appearance/ColorSchemeMode.swiftsystem / light / dark bridge
enum ColorSchemeMode: String, CaseIterable, Hashable, Identifiable {
    case system
    case light
    case dark

    var id: Self { self }

    var title: String {
        switch self {
        case .system: "Automatic"
        case .light: "Light"
        case .dark: "Dark"
        }
    }

    var nsAppearance: NSAppearance? {
        switch self {
        case .system: nil
        case .light: NSAppearance(named: .aqua)
        case .dark: NSAppearance(named: .darkAqua)
        }
    }
}

The toolbar picker uses icons and .glassEffect() because it is chrome. The settings picker uses labels and a conventional settings row because it is a preference panel. Both write to the same setting, so the app appearance stays coherent.

11. Complete reference pattern

For a new screen in this template, start with this structure. It keeps the top toolbar behavior intact, gives the detail view a native navigation title, and leaves room at the bottom without adding any visual effect.

Reusable destination scaffoldcopy this for ordinary scrolling screens
import SwiftUI

struct DashboardScreen: View {
    var body: some View {
        ScrollView {
            VStack(alignment: .leading, spacing: 14) {
                Text("Dashboard")
                    .font(.largeTitle)
                    .padding(.bottom, 4)

                // Cards, rows, charts, forms, or project-specific content.
            }
            .padding(24)
        }
        .scrollContentBackground(.hidden)
        .background(.clear)
        .scrollEdgeEffectStyle(.soft, for: .top)
        .safeAreaInset(edge: .bottom, spacing: 0) {
            Color.clear.frame(height: 56)
        }
        .navigationTitle("Dashboard")
    }
}

For a non-scrolling screen, omit the scroll modifiers. Do not wrap a non-scrolling screen in a dummy ScrollView just to satisfy the pattern. The contract applies to screens that actually scroll.

12. Step-by-step implementation checklist

  1. Create the app scene.Use a single main Window that renders ContentView. Persist appearance with AppStorage and apply it at the NSApp level.
  2. Install the split root.Use NavigationSplitView. Put the sidebar List(selection:) in the first column and the selected destination in the detail column.
  3. Attach toolbar chrome to the detail host.Place shared toolbar controls after detailView, not inside every destination screen.
  4. Keep toolbar background automatic.Use .toolbarBackgroundVisibility(.automatic, for: .windowToolbar). Do not hide the toolbar background when relying on the native top scroll edge.
  5. Make each scrolling destination transparent.Apply .scrollContentBackground(.hidden) and .background(.clear) to the destination ScrollView.
  6. Attach the top edge effect to the actual ScrollView.Use .scrollEdgeEffectStyle(.soft, for: .top) on the scroll view itself, not on a parent wrapper.
  7. Reserve bottom space without visual treatment.Add the clear bottom safeAreaInset only when the screen benefits from extra bottom breathing room.
  8. Register new screens in Screen.Add the case, title, SF Symbol, and destination. The sidebar updates automatically through Screen.allCases.

13. Failure modes and precise fixes

SymptomLikely causeFix
Toolbar becomes opaque or gets a hard edge.The scroll view was wrapped in an external compositing, mask, drawing, or effect layer; or the toolbar background was overridden.Keep the ScrollView native and apply .scrollEdgeEffectStyle(.soft, for: .top) directly to it.
Top soft edge disappears..toolbarBackgroundVisibility(.hidden, for: .windowToolbar) removed the material relationship the system effect resolves against.Restore .automatic on the detail host.
Only one destination has the correct toolbar behavior.The scroll recipe was added to one screen but not the others.Make the recipe part of the screen scaffold for every scrolling destination.
Last card feels cramped at the bottom.The bottom safe-area spacer was omitted or too small for the intended chrome.Add or tune Color.clear.frame(height: 56) in safeAreaInset(edge: .bottom).
Appearance picker changes the toolbar but not the app consistently.Appearance was applied locally instead of through the shared AppStorage / NSApp.appearance path.Keep the toolbar picker and Settings picker bound to the same stored ColorSchemeMode.
New files do not appear in Xcode.The generated project is stale.Run make project; do not manually patch the generated project file.

14. Verification protocol

Validate the template with both included destinations and at least one newly added destination. The point is not only to confirm that the current placeholders work, but to ensure the shell remains reusable after the first real app-specific screen is installed.

Visual cases

Cards, rows, long text, SF Symbols, selected sidebar row, toolbar segmented picker, light appearance, dark appearance, and system appearance.

Navigation cases

Initial selection, switching between screens, adding a third screen, opening Settings, changing appearance from Settings, and changing appearance from the toolbar.

  • The toolbar has no hard bottom border while content scrolls under it.
  • The top soft edge appears only near the top safe-area/toolbar region.
  • The bottom edge has no fade, tint, blur, or artificial darkening.
  • The last content item can scroll above the bottom edge with comfortable spacing.
  • Every scrollable destination uses the same scroll-edge recipe.
  • The toolbar picker and Settings picker stay synchronized.
  • Renaming the template, regenerating the project, running the app, and running tests all complete without manual project edits.

15. Extension points

The template is intentionally small, but it has clean seams for common app growth. Add features at the seam that owns the behavior rather than modifying the whole stack.

More destinations

Add cases to Screen. Keep destination layout decisions inside destination views.

More toolbar items

Add shared app-level actions in ContentView. Add screen-specific toolbar actions inside the destination only when they truly belong to that screen.

Bottom bars

Add a real bottom overlay only when the app has bottom chrome. Until then, keep the template's bottom edge as clear layout space.

Settings

Use the provided settings row components so future preferences keep the same macOS settings-window rhythm.

16. Glossary

Detail host
The right side of the NavigationSplitView, where the selected Screen.destination is rendered and where shared toolbar controls are attached.
Scroll-edge effect
The native SwiftUI treatment that softens content as it approaches a scroll view edge, used here only for the top toolbar boundary.
Toolbar background visibility
The setting that controls whether the macOS window toolbar background/material is available. This template uses .automatic.
Screen registry
The Screen enum that maps sidebar rows to destination views.
Bottom inset
A clear layout spacer inserted at the bottom safe area. It reserves scrollable breathing room but does not render a visual effect.
